# retro-codify スキル

## 設計思想

スキルは「AI 用の技術記事」である。良い知識単位の要件は 3 つ:

1. **問題が解決すること** — 抽象論ではなく具体的な課題に紐づく
2. **再現性があること** — 別のセッションでも同じ手順で再現できる
3. **新規性があること** — 既存スキルや CLAUDE.md に無い学びである

この 3 要件を満たさないものはスキル化しない（無理に増やさない）。

## 実行手順

1. 現在のセッションを振り返り、以下の問いに答える:
   - このセッションで**最も非自明だった判断**は何か？
   - **試行錯誤して解決した問題**はあったか？その鍵は何だったか？
   - **避けられた落とし穴**はあったか？次回も避けるべきか？
   - 同じことを別プロジェクトでやるなら、**何を知っていれば速いか**？

2. 抽出した学びを上記「3 要件」で自己審査する:
   - 解決した課題が明確か？（曖昧なら捨てる）
   - 手順として再現できるか？（できないなら捨てる）
   - 既存の `.claude/skills/` や CLAUDE.md と重複しないか？（重複なら捨てる）

3. 3 要件を通過した学びを `logs/retro-candidates.md` に追記する:

```markdown
## {YYYY-MM-DD HH:MM} セッション振り返り

### 候補1: {一行サマリー}
- **解決した課題**: ...
- **鍵となった判断/手順**: ...
- **再現手順の骨子**:
  1. ...
  2. ...
- **新規性**: 既存の〇〇には無い。〜という点が新しい
- **スキル化推奨度**: 高 / 中 / 低
```

4. **重要: ここでスキルを自動生成しない。** 候補を提示するだけに留める。
   実際の SKILL.md 化は人間が `/skill-builder` で承認したものだけ行う。

## やってはいけないこと

- 「直感的に」「自然と」といった曖昧な言葉で手順を書く
  → 直感の根拠を具体的な観察・条件として言語化すること
- 当たり前すぎる内容を候補にする（ノイズになる）
- 1 セッションから 5 個も 6 個も候補を出す（多くて 2〜3 個に絞る）
