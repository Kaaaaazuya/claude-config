# blog-draft スキル

## 前提確認

```bash
# 候補ファイルの存在確認
ls -la logs/retro-candidates.md 2>/dev/null || echo "retro-candidates.md なし"
```

`retro-candidates.md` がない場合は `/retro` を先に実行するよう案内して終了する。

## 実行手順

1. `logs/retro-candidates.md` を Read で読み込む

2. 候補を一覧表示してユーザーに選ばせる:

```
以下の候補があります。記事にするものを選んでください:

1. {候補1の一行サマリー}（推奨度: 高）
2. {候補2の一行サマリー}（推奨度: 中）
```

3. 選択された候補をもとに以下を決定する:
   - **タイトル**: 技術記事らしい日本語タイトル（「〜する方法」「〜で詰まったこと」形式が読まれやすい）
   - **slug**: タイトルを kebab-case 英語に変換（例: `claude-code-hook-tdd`）
   - **description**: 検索・OGP 向けの1〜2文の要約（日本語）
   - **tags**: 内容に合う技術キーワード（3〜5個、英小文字）

4. 以下のフォーマットで MDX ファイルを生成する:

````mdx
---
title: "{タイトル}"
description: "{description}"
pubDate: {YYYY-MM-DD}
tags: [{tag1}, {tag2}]
draft: true
---

## はじめに

{解決した課題・背景を1〜2段落で。「なぜこれを書くか」を明確に。}

## 状況

{問題が起きた状況・環境・前提条件を簡潔に。}

## 詰まったポイント

{試行錯誤した内容・罠になった点を具体的に書く。コードブロックを積極的に使う。}

```言語
{コード例}
```

## 解決策

{鍵となった判断・手順を再現可能な形で書く。}

```言語
{コード例}
```

## まとめ

{学んだこと・次に同じ状況に遭ったときのチェックリスト形式でまとめる。}

- [ ] ...
- [ ] ...
````

5. 出力先を確認する:
   - ユーザーの Astro プロジェクトパスが不明な場合は聞く
   - 既定: `src/content/blog/{YYYY-MM-DD}-{slug}.mdx`

6. ユーザーに最終確認を取ってから Write で保存する

## 品質チェック（保存前に確認）

- `draft: true` が設定されているか（誤公開防止）
- 個人情報・プロジェクト固有の情報（パス・社名・顧客名）が含まれていないか
- コードブロックに言語識別子が付いているか（MDX のシンタックスハイライトに必要）
- `description` が 80〜120 字程度か（短すぎると OGP が寂しい）

## やってはいけないこと

- `draft: false` で保存すること（必ず下書きで出す）
- retro-candidates.md の内容をそのままコピーすること（記事向けに書き直す）
- 1候補から複数記事を作ること（1候補 = 1記事）
