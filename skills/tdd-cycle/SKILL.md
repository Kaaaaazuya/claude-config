# tdd-cycle

TDD のフルサイクルをワンコマンドで回すスキル。
テスト作成から始めてドキュメント更新まで一貫して行う。

## 引数

```
/tdd-cycle <対象ファイルまたは機能の説明>
```

## ステップ0: テストフレームワーク自動検出

実装前にプロジェクトのテスト設定を確認する。以下の順で検出し、最初に見つかったものを使う。

| 検出ファイル | テストコマンド | カバレッジコマンド |
|---|---|---|
| `pyproject.toml` に `[tool.pytest]` または `pytest` 依存 | `pytest` | `pytest --cov` |
| `package.json` に `"jest"` | `npm test` / `jest` | `jest --coverage` |
| `package.json` に `"vitest"` | `vitest run` | `vitest run --coverage` |
| `Cargo.toml` | `cargo test` | `cargo tarpaulin` |
| `go.mod` | `go test ./...` | `go test -cover ./...` |

カバレッジは**テストライブラリ付属のレポーターを使う**。外部ツールを追加インストールしない。
検出できない場合はユーザーに確認する。

## ステップ1: テスト作成（Red）

- 対象の機能・インタフェースを確認してから、失敗するテストを書く
- テストを実行して **失敗することを確認する**（Red を確認せずに進まない）
- テストファイルの配置は既存のプロジェクト規約に従う

## ステップ2: 実装（Green）

- テストをパスする最小限の実装を書く
- 余分な機能・抽象化を加えない
- 実装後、テストを実行して **全パスを確認する**

## ステップ3: コードレビュー（subagent）

実装した差分を `code-reviewer` subagent に渡してレビューする。

```
Use the code-reviewer subagent to review the diff for bugs and edge cases.
Focus on: correctness of the implementation against the tests, unhandled edge cases.
```

- 指摘がなければステップ5へ
- 指摘があれば内容を確認し、対応するものだけ修正する（スタイル指摘は無視）

## ステップ4: 修正と再テスト（レビュー指摘がある場合）

- レビュー指摘を修正
- テストを再実行して全パスを確認
- 必要なら追加テストを書く

## ステップ5: カバレッジ計測

ステップ0で検出したコマンドでカバレッジを計測する。

- カバレッジレポートをサマリーとして表示する
- 明らかに未テストのパス（0% のブランチなど）があれば指摘する
- カバレッジ目標はプロジェクトの設定に従う（強制しない）

## ステップ6: ドキュメント更新

`sync-docs` スキルを実行してドキュメントを最新化する。

## ステップ7: サマリー提示

以下を出力して完了する。

```
## tdd-cycle 完了

- テスト: X件パス / X件スキップ
- カバレッジ: XX%（対象ファイル）
- レビュー所見: X件 → X件対応
- 更新ドキュメント: README.md / 開発計画.md（変更があった場合）
```

## 注意

- Red を確認せずにステップ2に進まない
- レビュー指摘をすべて対応する必要はない。correctness に関わるものだけ対応する
- カバレッジ100%を目指さない。意味のあるケースのテストを優先する
